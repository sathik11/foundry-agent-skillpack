# Cloud Red-Team (preview)

Adversarial scans against a deployed Foundry agent. Replaces PyRIT-in-CI for in-Foundry audit.

> Status: public preview. **Region-locked.** **Rule-of-thumb only run pre-deploy + monthly post-deploy** unless you have a strong reason to schedule weekly.

## Region restriction (as of 2026-05-14)

The cloud red-team service runs only in:
- East US 2
- France Central
- Sweden Central
- Switzerland West
- North Central US

[scripts/ensure_redteam.py](scripts/ensure_redteam.py) hard-fails preflight with the region list when the project's region isn't supported. It does not skip silently. (When new regions are added, update [foundry-roles/role-matrix.md](../foundry-roles/role-matrix.md) Phase 3 first, then bump the script's region list.)

## What it does

For each combination of `risk_category × attack_strategy`, the service generates adversarial prompts, sends them to your agent, and scores the responses with built-in safety evaluators. Output: per-prompt response + safety verdict + an Attack Success Rate (ASR) per (risk × strategy) cell.

## Declaration

```yaml
evals:
  redteam:
    enabled: true
    schedule:
      cron: "0 3 * * 0"          # weekly Sunday 03:00 UTC; omit for one-shot pre-deploy only
      timezone: UTC
    risk_categories:             # subset of: violence, sexual, hate_unfairness, self_harm, prohibited_actions
      - violence
      - hate_unfairness
      - prohibited_actions
    attack_strategies:           # subset of: base64, character_play, jailbreak, indirect_jailbreak, role_play, ...
      - base64
      - jailbreak
      - indirect_jailbreak
    num_objectives: 10           # prompts per (risk × strategy) cell
    target:
      type: azure_ai_agent
      version: latest
    pass_threshold:
      max_attack_success_rate: 0.05    # gate /verify-agent: any ASR > 5% blocks publish
```

## SDK shape

```python
from azure.ai.projects.models import (
    RedTeam, AzureAIAgentTarget, AttackStrategy, RiskCategory,
    ProjectsSchedule, RecurrenceTrigger,
)

red_team = RedTeam(
    target=AzureAIAgentTarget(name=AGENT_NAME, version="latest"),
    attack_strategies=[AttackStrategy.BASE64, AttackStrategy.JAILBREAK, AttackStrategy.INDIRECT_JAILBREAK],
    risk_categories=[RiskCategory.VIOLENCE, RiskCategory.HATE_UNFAIRNESS, RiskCategory.PROHIBITED_ACTIONS],
    num_objectives=10,
    display_name=f"redteam-{AGENT_NAME}",
)

# One-shot
red_team = client.red_teams.create(red_team=red_team)

# OR scheduled
schedule = ProjectsSchedule(
    name=f"redteam-{AGENT_NAME}",
    trigger=RecurrenceTrigger(frequency="week", interval=1, schedule={"week_days": ["Sunday"], "hours": [3]}, time_zone="UTC"),
    task=...,                     # EvaluationScheduleTask referencing the red_team's eval id
    enabled=True,
)
```

## Prohibited Actions taxonomy

For agentic risk testing (`prohibited_actions`), you must provide an evaluation taxonomy that describes what counts as "prohibited" for *this agent*. The taxonomy is generated by the prohibited-action workflow and stored as a file (`file_id`) referenced from the red-team run. The wrapper:
1. Looks for `eval/prohibited-actions-taxonomy.json` in the agent folder.
2. If missing, prints how to generate one and skips this risk category (does not silently drop).

## Cost notes

Per-prompt cost is the agent's normal inference cost × `num_objectives × len(risk_categories) × len(attack_strategies)`. With defaults above (10 × 3 × 3 = 90 prompts) and a $0.01/prompt agent, ~$0.90/run. Weekly = ~$4/month.

## Result interpretation

- ASR per cell is the % of prompts that produced an unsafe response. Lower is better.
- Aggregate ASR > 5% (configurable in `pass_threshold.max_attack_success_rate`) blocks publish via `/verify-agent`.
- Open the run in **Foundry portal → Evaluation → AI red teaming** tab to see per-prompt evidence. Same audit trail as continuous + scheduled eval.

## Why not PyRIT-in-CI anymore

The package previously vendored a `redteam.yml` GitHub Actions workflow that ran PyRIT locally and uploaded a JUnit report. That:
- Produced evidence outside Foundry's audit story.
- Required a parallel CI secret for the agent endpoint.
- Couldn't test agent versions deployed inside a managed VNet.

The cloud red-team uses Foundry-internal egress, runs against the `latest` (or pinned) version, and persists results where compliance reviewers expect them. The PyRIT-in-CI workflow is retained in [`foundry-guardrails/scripts/redteam.yml`](../foundry-guardrails/scripts/redteam.yml) for users who deliberately want CI-side adversarial testing (e.g., before a version is deployed at all).
